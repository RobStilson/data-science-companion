# Data Science Companion — SPEC.md

## 1. Objective

Build a conversational AI agent that guides users through exploratory data analysis (EDA)
of uploaded flat files. The agent produces structured reports, interactive visualizations,
and downloadable Python and R scripts that reproduce everything it did.

**Target users:** Mixed — data scientists / ML engineers who want depth, and business
analysts who need plain-language summaries alongside technical output.

**Success criteria:** A user can upload a CSV or Excel file, receive a complete EDA
(data dictionary through skewness), ask for correlation analysis against an outcome
variable, request visualizations, and download fully-runnable .py and .R export files —
all within a single Chainlit chat session.

---

## 2. Core Features & Acceptance Criteria

### F-01 File Upload
- User uploads a CSV or Excel (.xlsx / .xls) file via Chainlit's file upload widget.
- Agent loads the file into a pandas DataFrame and confirms shape, filename, and preview
  (first 5 rows).
- Acceptance: Unsupported formats return a clear error message; files >100 MB return a
  size warning before loading.

### F-02 Data Dictionary
- Produces a table with: column name, inferred dtype, sample values (3), null count,
  null %, and a plain-language description generated by the LLM.
- Acceptance: All columns present; LLM descriptions are <= 1 sentence each.

### F-03 Missing Data Report
- Lists every column with nulls: count, percentage, and a severity label
  (Low < 5%, Medium 5-20%, High > 20%).
- Includes an overall dataset completeness percentage.
- Acceptance: Columns with 0 nulls are excluded from the detail table but mentioned in
  the summary.

### F-04 Descriptive Statistics
- Runs `df.describe()` for numeric columns with additional callouts:
  - Columns where std > mean (high dispersion)
  - Columns where min == max (zero variance)
  - Potential outliers flagged via IQR method (values beyond Q1 - 1.5*IQR or Q3 + 1.5*IQR)
- Plain-language summary of notable findings.
- Acceptance: Callouts section is present even when nothing notable is found
  ("No anomalies detected").

### F-05 Categorical vs. Numerical Classification
- **Categorical list:** columns with dtype object/category, or numeric columns with
  <= 20 unique values flagged as likely categorical.
- **Numerical list** split into:
  - Discrete: numeric columns with < 10 unique values
  - Continuous: numeric columns with >= 10 unique values
- Acceptance: Each column appears in exactly one list; classification reasoning shown.

### F-06 Correlation Analysis (Optional — User-Triggered)
- Agent asks: "Is there an outcome variable you'd like to analyse correlations for?"
- If yes, user names the column; agent:
  - Auto-selects correlation method based on data types:
    - Pearson: continuous numeric predictor + continuous numeric outcome
    - Spearman: skewed or ordinal predictor
    - Point-Biserial: binary (0/1) outcome
  - Returns a ranked correlation table (predictor -> outcome) with method noted per row.
  - Highlights top 5 positive and top 5 negative correlations.
- Acceptance: Method selection logic is shown to user; unknown outcome column returns
  a helpful error.

### F-07 Data Visualization
- Agent prompts: "Which column(s) would you like to visualise, and what type of chart?
  Here are my suggestions: [list based on detected dtypes]."
- Supported chart types: histogram, box plot, scatter plot, bar chart, heatmap
  (correlation matrix), line chart, pair plot.
- Renders interactive Plotly chart in Chainlit chat.
- Multiple visualizations supported per session.
- Acceptance: Unknown column names return a clear error; unsupported chart types prompt
  the user to choose from the supported list.

### F-08 Skewness Report
- Computes skewness for all numeric columns.
- Labels: Symmetric (|skew| < 0.5), Moderate (0.5-1.0), High (> 1.0).
- Suggests transformations for highly skewed columns (log, sqrt, Box-Cox).
- Acceptance: Results presented as a sorted table; plain-language summary included.

### F-09 Code Export
- At any point the user can request "export" or "download code."
- Agent generates:
  - `analysis_export.py` — fully runnable Python script using pandas + matplotlib
    reproducing all analyses run so far in the session.
  - `analysis_export.R` — equivalent R script using tidyverse + ggplot2.
- Files are offered as Chainlit downloadable file attachments.
- Acceptance: Scripts are self-contained (include imports and inline data path
  placeholder); they run without modification beyond setting the file path.

## 3. Tech Stack

| Layer | Choice | Notes |
|---|---|---|
| Language | Python 3.11+ | |
| Agent framework | LangGraph | Graph-based agent; each analysis step is a node |
| LLM integration | LangChain (provider-agnostic) | Configured via `LLM_PROVIDER` + `LLM_MODEL` env vars |
| Supported LLM providers | Anthropic, OpenAI, Ollama | Abstracted behind a factory in `llm/factory.py` |
| Frontend | Chainlit | File upload, chat UI, downloadable file attachments |
| Data layer | pandas, openpyxl | DataFrame operations; openpyxl for Excel |
| Visualizations (UI) | Plotly | Interactive charts rendered in Chainlit |
| Visualizations (export) | Matplotlib / Seaborn | Static charts embedded in exported Python script |
| Statistics | scipy, statsmodels | Skewness, correlation tests, IQR outliers |
| Config | python-dotenv | `.env` file for API keys and provider selection |
| Testing | pytest + pytest-asyncio | Unit tests per analysis module; integration test for full EDA flow |
| Linting | ruff | Fast linter + formatter |

## 5. Code Style

- No comments unless the WHY is non-obvious.
- Type hints on all function signatures.
- Each `agent/nodes/*.py` file exposes a single async function:
  `async def run(state: AgentState) -> AgentState`.
- LLM calls go through `llm/factory.py` only — never instantiate a model directly in a node.
- Prompts live in `prompts/` as Markdown files loaded at startup — not hardcoded strings.
- Output to Chainlit via `cl.Message` only — no `print()` statements in agent code.
- `ruff` must pass with zero warnings before any commit.
- Max line length: 100 characters.

---

## 6. Testing Strategy

- **Unit tests** for each analysis node: input is a small test DataFrame from
  `tests/fixtures/`; output is validated for structure (columns, keys) and edge cases
  (all-null column, single-row dataset, numeric-only dataset).
- **LLM calls are mocked** in unit tests using `unittest.mock.patch`.
- **Integration test** (`tests/test_integration.py`): runs the full LangGraph against
  `sample.csv` without mocking and asserts all report sections are present in output.
- **Export test**: asserts generated `.py` and `.R` files are non-empty and contain
  expected function names / library imports.
- Target: >= 80% coverage on `agent/` and `utils/`.

## 7. Boundaries

### Always Do
- Present both a plain-language summary and a technical table for every analysis.
- Show which correlation method was chosen and why.
- Confirm file loaded successfully (shape + preview) before running any analysis.
- Offer the export option after every major analysis step.

### Ask the User First
- Before running correlation analysis (outcome variable must be named by user).
- Before generating a visualization (confirm columns + chart type).
- If a file exceeds 100 MB (warn about potential slowness, ask to proceed).
- If Excel file has multiple sheets (ask which sheet to use).

### Never Do
- Modify or overwrite the uploaded file.
- Store uploaded data beyond the current session.
- Hardcode API keys or provider names in source files.
- Make LLM calls for purely computational tasks (statistics, dtype detection) — LLM
  is used only for natural language description and interpretation.

## 8. Development Workflow

```
spec -> plan -> build -> test -> review -> simplify -> ship
```

Each phase produces a deliverable before the next begins:
- **spec**     -> this file (SPEC.md)
- **plan**     -> PLAN.md with ordered build tasks
- **build**    -> working implementation, one node at a time
- **test**     -> pytest suite green, >= 80% coverage
- **review**   -> code-review skill; address all findings
- **simplify** -> simplify skill; remove unnecessary complexity
- **ship**     -> requirements.txt locked, .env.example complete, README written
